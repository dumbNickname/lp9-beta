# PRD-07 — Supabase bootstrap (GitHub integration + first migration)

## Goal

Wire the local repo to the already-connected Supabase project via the
`supabase/` directory at repo root, prove the GitHub integration runs
migrations end-to-end with a no-op migration, and document the
branch-driven workflow so every future PRD that touches schema
follows the same pattern.

## Scope

**In:**
- `supabase/` directory at repo root, initialized via
  `supabase init` (working directory `.` matches what the owner set
  in the Supabase dashboard's GitHub Integration).
  - `supabase/config.toml` (project config; commit).
  - `supabase/migrations/` directory.
  - `supabase/seed.sql` empty file (placeholder for future seed data
    on preview branches; per Supabase docs, `seed.sql` is what
    populates preview branch DBs).
- First migration `supabase/migrations/0000_init.sql` that's a no-op
  (just a `-- placeholder` comment) — purely to verify the GitHub
  integration's migration pipeline works end-to-end.
- `supabase/README.md` explaining:
  - The branch-driven workflow per `DESIGN.md` §16e and §16f.
  - **How to add a new migration:** `supabase migration new
    <slug>`, edit the file, commit on a feature branch, open PR,
    Supabase auto-creates a preview branch, verify, merge to
    `master`, the prod DB gets the migration automatically.
  - The rule: schema changes always go via migration files,
    never click-ops in the Supabase dashboard.
  - Where the project URL + anon key come from (dashboard → API),
    and that they go into local `.env` (per PRD-06 hygiene).
  - That **no personal access token, no DB password is needed**
    for the branch-driven flow.
- `.env.example` (created in PRD-06) confirmed to list only
  `VITE_SUPABASE_URL` and `VITE_SUPABASE_ANON_KEY`.
- A short paragraph in the root `README.md` Local-setup section
  pointing at `supabase/README.md` for DB workflow.

**Out:**
- Real schema (deferred to Phase 1+ PRDs).
- Supabase Edge Functions (deferred until needed).
- Anonymous sign-in toggle in the dashboard (Phase 1 PRD).
- Branch-protection rules on `master` requiring the Supabase Preview
  check (deferred to PRD-09 deploy workflow, where all branch
  protection rules are configured together).
- Local `supabase start` Docker stack setup. Useful for offline
  iteration, but not required for the branch-driven flow. Cover in
  a later PRD only if a contributor asks.

   passed).
2. `supabase/config.toml` exists, is committed, and contains no
   secrets (no API keys, no DB passwords). The generated default is

3. `supabase/migrations/0000_init.sql` exists with no-op content.
4. `supabase/seed.sql` exists (empty file is fine).
5. `supabase/README.md` documents:
   - The branch-driven workflow.

   - Where to find the project URL / anon key.
   - That no PAT or DB password is needed.
6. **End-to-end pipeline check:** create a throwaway feature branch,
   push it to `beta`, open a PR. Within ~2 minutes the Supabase
   GitHub integration should:
   - Post a status check on the PR (`Supabase Preview` or similar).
   - Create a preview branch in the Supabase dashboard.
   - Apply `0000_init.sql` to the preview DB without error.
7. Clean up the throwaway branch after verifying.
